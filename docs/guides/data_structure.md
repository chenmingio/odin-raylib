数据结构选择

很高兴你感兴趣！这些数据结构的选择往往取决于访问模式和硬件特性（主要是缓存）。

  游戏/系统编程常见的数据结构

  ---
  1. SOA vs AOS（最重要的概念之一）

  // AOS: Array of Structures（常规写法）
  entities: [1000]struct {
      pos:    V3,
      vel:    V3,
      health: f32,
      name:   string,
  }

  // SOA: Structure of Arrays
  entities: struct {
      pos:    [1000]V3,
      vel:    [1000]V3,
      health: [1000]f32,
      name:   [1000]string,
  }
  ┌──────────────────────┬────────────────────┬───────────────┐
  │         方面         │        AOS         │      SOA      │
  ├──────────────────────┼────────────────────┼───────────────┤
  │ 访问单个实体所有字段 │ ✅ 快（连续）      │ ❌ 慢（跳跃） │
  ├──────────────────────┼────────────────────┼───────────────┤
  │ 批量处理单个字段     │ ❌ 慢（stride 大） │ ✅ 快（连续） │
  ├──────────────────────┼────────────────────┼───────────────┤
  │ SIMD 友好            │ ❌                 │ ✅            │
  ├──────────────────────┼────────────────────┼───────────────┤
  │ 代码直观性           │ ✅                 │ ❌            │
  └──────────────────────┴────────────────────┴───────────────┘
  适用场景：
  - 物理引擎只更新 pos += vel？用 SOA
  - 渲染时需要 pos + texture + color 一起用？用 AOS

  2. Slot Map / Generational Index

  解决"实体被删除后，旧引用变成野指针"的问题。

  SlotMap :: struct {
      data:       [dynamic]Entity,
      generation: [dynamic]u32,      // 每个槽的"代数"
      free_list:  [dynamic]u32,
  }

  Handle :: struct {
      index:      u32,
      generation: u32,  // 创建时的代数
  }

  get :: proc(map: ^SlotMap, handle: Handle) -> ^Entity {
      // 代数不匹配 = 这个槽已经被复用了，旧 handle 失效
      if map.generation[handle.index] != handle.generation {
          return nil
      }
      return &map.data[handle.index]
  }
  ┌──────────┬─────────────────┬────────────────────┐
  │   方面   │     裸指针      │      Slot Map      │
  ├──────────┼─────────────────┼────────────────────┤
  │ 安全性   │ ❌ 野指针风险   │ ✅ 自动失效        │
  ├──────────┼─────────────────┼────────────────────┤
  │ 访问速度 │ ✅ O(1)         │ ✅ O(1) + 一次比较 │
  ├──────────┼─────────────────┼────────────────────┤
  │ 内存稳定 │ ❌ 可能 realloc │ ✅ 索引不变        │
  └──────────┴─────────────────┴────────────────────┘
  用于：ECS 系统、对象池、网络同步（handle 可序列化）

  3. Ring Buffer（循环缓冲区）

  写入位置 ↓
  [4] [5] [6] [ ] [ ] [1] [2] [3]
                       ↑ 读取位置

  写满后覆盖最旧的数据

  RingBuffer :: struct($T: typeid, $N: int) {
      data:  [N]T,
      read:  u32,
      write: u32,
  }
  ┌───────────────┬──────────────┬──────────────────────────┐
  │     方面      │   动态数组   │       Ring Buffer        │
  ├───────────────┼──────────────┼──────────────────────────┤
  │ 内存分配      │ 可能 realloc │ ❌ 永不分配              │
  ├───────────────┼──────────────┼──────────────────────────┤
  │ 保留历史      │ 全部         │ 只保留最近 N 个          │
  ├───────────────┼──────────────┼──────────────────────────┤
  │ 生产者/消费者 │ 需要锁       │ 可以无锁（单生产单消费） │
  └───────────────┴──────────────┴──────────────────────────┘
  用于：音频缓冲、输入历史、日志、网络包队列

  4. Intrusive vs Non-intrusive 容器

  // Non-intrusive: 容器持有数据的副本或指针
  List :: struct {
      nodes: [dynamic]struct {
          data: ^Entity,  // 指向外部
          next: u32,
      },
  }

  // Intrusive: 链表节点嵌入数据本身
  Entity :: struct {
      pos:  V3,
      // ... 其他字段
      next_in_chunk: ^Entity,  // 链表指针嵌入实体
  }
  ┌──────────────────┬──────────────────────┬────────────────────────┐
  │       方面       │    Non-intrusive     │       Intrusive        │
  ├──────────────────┼──────────────────────┼────────────────────────┤
  │ 数据结构独立性   │ ✅ Entity 不知道容器 │ ❌ Entity 必须包含节点 │
  ├──────────────────┼──────────────────────┼────────────────────────┤
  │ 内存分配         │ 需要额外分配节点     │ ❌ 无额外分配          │
  ├──────────────────┼──────────────────────┼────────────────────────┤
  │ 一个元素多个容器 │ ✅ 容易              │ ❌ 需要多个 next 指针  │
  ├──────────────────┼──────────────────────┼────────────────────────┤
  │ 缓存局部性       │ ❌ 差                │ ✅ 好                  │
  └──────────────────┴──────────────────────┴────────────────────────┘
  Linux 内核大量使用 intrusive list，因为零额外分配。

  5. Open Addressing vs Chaining（哈希表）

  Chaining（你代码里的 chunk_hash）:
  bucket[0] -> Chunk -> Chunk -> Chunk
  bucket[1] -> Chunk
  bucket[2] -> nil

  Open Addressing（线性探测）:
  bucket: [Chunk, Chunk, empty, Chunk, empty, ...]
  冲突时往后找空位
  ┌────────────┬─────────────┬─────────────────────┐
  │    方面    │  Chaining   │   Open Addressing   │
  ├────────────┼─────────────┼─────────────────────┤
  │ 缓存局部性 │ ❌ 链表跳跃 │ ✅ 连续内存         │
  ├────────────┼─────────────┼─────────────────────┤
  │ 删除操作   │ ✅ 简单     │ ❌ 需要墓碑标记     │
  ├────────────┼─────────────┼─────────────────────┤
  │ 负载因子   │ 可以 >1     │ 必须 <1（通常 0.7） │
  ├────────────┼─────────────┼─────────────────────┤
  │ 内存使用   │ 额外指针    │ 可能有空洞          │
  └────────────┴─────────────┴─────────────────────┘
  Robin Hood Hashing：Open Addressing 的改进，让"穷人"（探测距离长的）抢"富人"的位置，减少最坏情况。

  7. Free List（你代码里有）

  // 复用已分配的内存块
  World :: struct {
      first_free: ^WorldEntityBlock,  // 空闲链表头
  }

  // 分配：从 free list 取，没有再真正分配
  // 释放：放回 free list，不真正释放

  优点：避免频繁 malloc/free，减少内存碎片

  选择数据结构的思考框架

  1. 访问模式是什么？
     - 遍历全部？→ 连续数组、SOA
     - 点查找？→ 哈希表
     - 范围查询？→ 树、空间哈希

  2. 数据量多大？
     - 小（<100）→ 数组暴力扫描可能更快
     - 大（>10000）→ 需要合适的索引结构

  3. 修改频率？
     - 频繁增删 → 链表、slot map
     - 几乎不变 → 排序数组 + 二分

  4. 缓存是否重要？
     - 热点路径 → 优化缓存局部性
     - 冷路径 → 简单优先
