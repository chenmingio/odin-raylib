Game内存设计


传统的malloc是手动分配+手动释放。分配的时候allocator会自己找到合适的内存位置后再插入，但还需要手动释放。找的时候需要找到之前释放的空位，找和释放的动作会影响性能。

准确的说：
malloc 不是“插入”，而是：在 free list / bin 里找合适块。可能 split，可能合并（free 的时候）。这些操作有分支，有 cache miss，有锁（多线程），有碎片问题。对实时系统非常不友好。

Arena模式是指定一片内存，只管分配，而且是连续分配，不用找“空位”。
相比之下，Arena对于游戏开发，内存分配快，不需要寻找，可以批量释放。

坏处是内存得不到释放，可能很快满了。

一个思路就是：分开“常驻”和“临时”需求，分层处理。

对于临时的内存需求，把他们都放到Arena里，然后批量释放所有的。

 临时 Arena 的一种用法——嵌套临时分配：

  // 保存当前位置
  mark := mem.arena_temp_begin(&temp_arena)

  // 做一些临时分配...
  temp_buffer := make([]u8, 1024, temp_alloc)
  // 用完...

  // 恢复到之前的位置（局部释放）
  mem.arena_temp_end(mark)

  这样可以在不清空整个 Arena 的情况下释放局部临时数据。

Odin原生支持多种内存分配器👍


编码难点

C语言里，用alloc分配一大块memory以后，就可以把memory的起点（根据自己的约定）转换为gameState: game_state *GameState = (game_state *)Memory->PermanentStorage;

Odin里，先为arena allocator用heap分配一段memory，然后需要用new来分配空间（game_state = new(GameState)）。注意不能像c一样直接使用memory的指针来转换，因为odin的memory的开头有元数据，相当于memory内部是实现，不能直接打开使用。
